\chapter{Implementação da Solução}
\label{cap:implementação}

Neste capítulo descreve-se a implementação do backoffice multiaplicação e demonstra-se de que forma o desenho apresentado no Capítulo~\ref{cap:analise_desenho} foi concretizado no código e na interface. A organização segue a separação arquitetural já definida: primeiro apresenta-se a implementação do \textit{back-end}, onde a identidade autenticada é integrada com autorização, regras de domínio, persistência, auditoria e comunicação externa; depois apresenta-se a implementação do \textit{front-end}, onde essas decisões se tornam visíveis para o operador. No final, descrevem-se os testes e a avaliação da solução, distinguindo a validação manual realizada pelo autor dos testes automatizados conduzidos no processo de \gls{QA} da empresa.

%----------------------------------------------------------------------------------------

\section{Implementação do back-end}
\label{sec:impl_backend}

O \textit{back-end} materializa a vista de componentes e a vista de processo apresentadas nas Figuras~\ref{fig:c4_level3_be}, \ref{fig:c4_level3_implementation} e \ref{fig:backend_process_view}. A regra central é a mesma descrita no Capítulo~\ref{cap:analise_desenho}: o pedido entra pela \gls{API}, é encaminhado por rotas, adaptado por controladores, processado por serviços de domínio e persistido através de repositórios. Esta organização permite que a interface não decida sozinha operações sensíveis e que o núcleo de autorização do backoffice seja reutilizável por futuras aplicações.

\subsection{Estrutura por camadas e injeção de dependências}
\label{sec:ambiente_infraestrutura}

O repositório está organizado em duas pastas raiz independentes: \texttt{be/} para o \textit{back-end} e \texttt{fe/} para o \textit{front-end}. No \textit{back-end}, a estrutura segue a separação descrita no Nível 3 da arquitetura: rotas, controladores, serviços, repositórios, modelos de domínio e objetos de transferência de dados. O servidor arranca com \texttt{ts-node} em desenvolvimento e compila para JavaScript em produção.

A injeção de dependências é configurada no ficheiro \texttt{be/src/loaders/index.ts}, que regista repositórios, serviços e controladores no contentor do TypeDI antes de o Express montar as rotas. Esta opção corresponde à vista de implementação do \textit{back-end}: as dependências são resolvidas no arranque e não durante cada pedido, reduzindo acoplamento entre módulos e evitando que controladores conheçam detalhes concretos de persistência. Assim, um controlador de campanhas, categorias ou verificações KYC/KYB trabalha sobre serviços e interfaces já registados, mantendo a regra de negócio fora da camada HTTP.

A entrega para o ambiente de desenvolvimento está parcialmente automatizada com Bitbucket Pipelines. No \textit{back-end}, a \textit{pipeline} constrói uma imagem Docker, autentica-se no Amazon ECR através da AWS CLI, aplica uma etiqueta imutável baseada em \textit{timestamp} UTC e publica tanto essa versão como a etiqueta \texttt{latest}. Esta forma de entrega acompanha a vista física do Capítulo~\ref{cap:analise_desenho}: o código da \gls{API} é empacotado como unidade executável e disponibilizado na infraestrutura AWS sem depender do ciclo de publicação do \textit{front-end}.

\subsection{Integração da identidade autenticada e contexto de autorização}
\label{sec:autenticacao_impl}

A autenticação é gerida pelo AWS Cognito com validação de tokens JWT e constitui uma dependência transversal do sistema. A contribuição descrita neste relatório começa no passo seguinte: ligar essa identidade autenticada ao modelo interno de autorização do backoffice. Depois de validar o token, o \textit{middleware} \texttt{cognitoAuth.ts} localiza o operador no backoffice, confirma o estado da conta, resolve os perfis guardados em \texttt{roleIds} e calcula o acesso efetivo. O resultado é anexado a \texttt{req.auth}, incluindo \texttt{roles}, \texttt{permissions}, \texttt{permissionsByApplication}, \texttt{appsAccessible} e \texttt{isSuperAdmin}.

A Listagem~\ref{lst:effective_access} apresenta o ponto em que a autorização passa a depender simultaneamente da aplicação associada ao perfil e da aplicação associada à permissão. Esta listagem é a implementação direta da regra descrita no modelo de permissões: perfis aplicacionais só recebem permissões da sua própria aplicação, enquanto perfis de backoffice seguem regras globais próprias.

\begin{lstlisting}[float, caption={Excerto do cálculo do acesso efetivo por aplicação}, label={lst:effective_access}]
const backofficeRole =
  roles.find(role => role.application === "backoffice") ?? null;
const isSuperAdmin =
  roles.some(role => role.name === SUPER_ADMIN_ROLE_NAME);

if (isSuperAdmin) {
  allPermissions.forEach(permission =>
    addPermission(permissionsByApplicationSets, permission),
  );
} else if (backofficeRole) {
  allPermissions
    .filter(permission => permission.application !== "backoffice")
    .forEach(permission =>
      addPermission(permissionsByApplicationSets, permission),
    );

  backofficeRole.permissions.forEach(rolePermission => {
    const permission = permissionByName.get(rolePermission.name);
    if (permission?.application === "backoffice") {
      addPermission(permissionsByApplicationSets, permission);
    }
  });
} else {
  roles.forEach(role => {
    role.permissions.forEach(rolePermission => {
      const permission = permissionByName.get(rolePermission.name);
      if (permission?.application === role.application) {
        addPermission(permissionsByApplicationSets, permission);
      }
    });
  });
}

const permissionsByApplication =
  toPlainPermissions(permissionsByApplicationSets);
\end{lstlisting}

A Listagem~\ref{lst:effective_access} evidencia duas decisões importantes para a escalabilidade. Primeiro, a matriz de permissões é calculada no servidor e devolvida já agrupada por aplicação, evitando que o cliente reconstrua regras a partir do token Cognito. Segundo, uma nova aplicação não exige uma nova estratégia de autorização: basta registar as suas permissões, criar perfis associados e configurar as páginas correspondentes. Esta implementação concretiza os requisitos RF02, RF03 e RF07.

\subsection{Proteção de endpoints}
\label{subsec:auth_step4}

Depois de resolvido o contexto do operador, cada endpoint sensível volta a validar permissões no servidor. Esta segunda barreira concretiza a separação entre experiência de interface e segurança efetiva: o \textit{front-end} pode esconder ações indisponíveis, mas a \gls{API} continua a decidir se o pedido deve ser executado. O padrão repete-se ao longo das rotas: a listagem de campanhas exige \texttt{view\_campaigns} ou permissões equivalentes, a edição de categorias exige \texttt{edit\_categories}, as verificações KYC/KYB exigem \texttt{view\_kyc} ou \texttt{view\_kyb}, e os logs exigem permissões administrativas próprias.

O excerto da Listagem~\ref{lst:require_permission} mostra o núcleo desta validação. A regra é conservadora: sem contexto autenticado, a resposta é \texttt{401}; com contexto autenticado mas sem permissões suficientes, a resposta é \texttt{403}; apenas pedidos autorizados avançam para o controlador.

\begin{lstlisting}[float, caption={Validação de permissões no middleware \texttt{requirePermission}}, label={lst:require_permission}]
export function requirePermission(...required) {
  return (req, res, next) => {
    const auth = req.auth;
    if (!auth) {
      res.status(401).json({ message: "Unauthorized" });
      return;
    }

    if (auth.isSuperAdmin) {
      next();
      return;
    }

    const hasPermission = required.some((p) =>
      auth.permissions.includes(p),
    );

    if (!hasPermission) {
      res.status(403).json({
        message: "Forbidden: insufficient permissions",
      });
      return;
    }

    next();
  };
}
\end{lstlisting}

Quando um pedido é recusado por falta de permissões, o servidor infere a aplicação a partir da rota, regista o evento \texttt{APP\_ACCESS\_FAILED} e aplica uma janela curta de deduplicação para limitar registos repetidos sobre o mesmo recurso. Esta decisão liga a implementação ao requisito RF13: acessos negados relevantes também fazem parte da rastreabilidade operacional.

\subsection{Perfis, permissões e configuração administrativa}
\label{subsec:impl_roles_permissions}

A gestão de operadores, perfis e permissões concretiza os requisitos RF03, RF05 e RF07. No modelo atual, cada operador pode ter no máximo um perfil por aplicação. Os perfis de backoffice são globais e exclusivos, pelo que um operador com esse tipo de perfil não acumula perfis aplicacionais. Esta regra impede configurações ambíguas e mantém clara a diferença entre administração transversal e operação de uma aplicação específica.

A Listagem~\ref{lst:role_permission_validation} mostra duas validações servidor-side que impedem estados incoerentes: duplicar perfis da mesma aplicação num operador e associar permissões de uma aplicação a um perfil de outra.

\begin{lstlisting}[float, caption={Validações de perfis e permissões por aplicação}, label={lst:role_permission_validation}]
const applications = new Set<string>();
for (const role of roles) {
  if (applications.has(role.application)) {
    throw this.buildError(
      "ROLE_APPLICATION_DUPLICATE",
      "A user can only have one role per application.",
    );
  }
  applications.add(role.application);
}

if (
  roles.some(role => role.application === "backoffice") &&
  roles.length > 1
) {
  throw this.buildError(
    "BACKOFFICE_ROLE_EXCLUSIVE",
    "Backoffice roles are global and cannot be combined " +
      "with application roles.",
  );
}

private assertPermissionMatchesRoleApplication(
  application: RoleApplication,
  permission: Permission,
): void {
  if (permission.application !== application) {
    throw this.buildError(
      "PERMISSION_APPLICATION_MISMATCH",
      `Permission "${permission.name}" belongs to ` +
        `"${permission.application}" and cannot be assigned to ` +
        `a "${application}" role.`,
    );
  }
}
\end{lstlisting}

Estas validações demonstram que a coerência do modelo não depende apenas da interface. Mesmo que um pedido manual tente atribuir uma permissão da JustCauses a um perfil de backoffice, ou combinar um perfil global com perfis aplicacionais, a gravação é rejeitada no servidor. Esta regra concretiza o modelo de dados administrativo apresentado na Figura~\ref{fig:backoffice_data_model}: perfis, permissões, \textit{page gates} e operadores pertencem ao núcleo administrativo do backoffice, persistido em DynamoDB.

\subsection{Auditoria e atividade dos operadores}
\label{subsec:impl_audit}

Todas as operações de escrita sobre entidades críticas geram um registo de auditoria no AWS DynamoDB. Cada registo contém o identificador do ator, o tipo e identificador da entidade afetada, a ação executada, a marca temporal em ISO 8601 e metadados específicos da ação, como razão, estado anterior ou estado novo. Os registos são imutáveis: não existem operações de atualização ou eliminação no \texttt{AuditLogRepo}.

A página de logs consulta estes registos com filtros por data, operador e tipo de ação. Para além das operações de negócio, o sistema regista tentativas falhadas de acesso a aplicações e recusas por permissões insuficientes. Esta implementação encaixa no modelo de dados administrativo e na vista de processo de Nível 2: a \gls{API} executa a decisão e escreve auditoria no DynamoDB, mantendo a rastreabilidade separada dos dados operacionais da JustCauses.

A funcionalidade \textbf{Activity} agrega, para um operador específico, eventos de \textit{audit logs} e \textit{audit trail}. Esta agregação responde ao objetivo OE1.4, porque permite analisar ações administrativas, acessos tentados, resultado, aplicação afetada e metadados sem obrigar o leitor ou o operador a consultar fontes separadas. O SD detalhado da Figura~\ref{fig:sd_user_activity_full}, no Apêndice~\ref{app:sd_detalhados}, expande esta interação técnica.

\subsection{Fluxos operacionais da JustCauses no servidor}
\label{sec:modulos_ojc}

Os módulos da JustCauses concretizam os objetivos OE2.1 a OE2.5 usando a mesma cadeia de responsabilidades do \textit{back-end}. A \gls{API} não trata estes módulos como exceções ao desenho: campanhas, categorias, analytics e verificações KYC/KYB entram por rotas protegidas, são processados por controladores e serviços próprios, acedem à persistência adequada e registam auditoria quando existe decisão administrativa.

Na revisão de campanhas, a listagem operacional e a \textit{thread} de revisão são tratadas separadamente. A listagem permite localizar campanhas e perceber o seu estado; a \textit{thread} concentra a decisão sobre uma submissão concreta. A Listagem~\ref{lst:campaign_revision_flow} apresenta um excerto representativo do fluxo de revisão: o controlador chama o serviço de domínio, o serviço aplica a decisão e notifica o criador, e o controlador regista a auditoria da operação.

\begin{lstlisting}[float, caption={Excerto do fluxo de revisão de campanha}, label={lst:campaign_revision_flow}]
const detail = await this.service.moderateThread(
  threadId,
  action,
  actor.adminUserId,
  bodyMessage,
);
if (!detail) {
  return res.status(404).json({
    message: "Campaign revision thread not found",
  });
}

this.auditLogService.log({
  ...actor,
  action: buildRevisionAuditAction(action),
  targetType: "campaign",
  targetId: detail.campaignId,
  targetLabel: detail.campaign.title,
  details: buildRevisionAuditDetails(action, detail, bodyMessage),
});

await this.repo.applyModerationAction({
  context,
  action,
  message,
  reviewedByAccountId,
});

await this.notifyCreator(context, action, message);
\end{lstlisting}

Este excerto demonstra a aplicação do padrão de camadas do Capítulo~\ref{cap:analise_desenho}. O controlador adapta o pedido HTTP e conhece o ator; o serviço aplica a regra de domínio e delega a persistência no repositório; a auditoria fica ligada ao alvo e à ação; a notificação ao criador é consequência da decisão tomada. O SD detalhado da Figura~\ref{fig:sd_campaign_revision_full}, no Apêndice~\ref{app:sd_detalhados}, apresenta este fluxo com os participantes técnicos envolvidos.

No módulo de categorias, a \gls{API} expõe endpoints de listagem, criação, edição e remoção em \path{/api/ojc/categories}. A criação e a edição validam unicidade de nome e \textit{slug}, devolvendo \texttt{409 Conflict} em caso de duplicado. Todas as operações de escrita são registadas em auditoria. No módulo de \textit{analytics}, a \gls{API} expõe \texttt{GET /api/ojc/analytics} com filtros de período e devolve agregações calculadas sobre a base relacional da JustCauses, incluindo evolução temporal, distribuição por categoria e campanhas com maior volume angariado.

A verificação KYC/KYB concretiza diretamente RF11 e OE2.5. No fluxo KYC, o endpoint \path{/api/ojc/kyc} é protegido por \texttt{view\_kyc} e devolve submissões filtráveis por estado, pesquisa, país, tipo de documento, comparação de dados e atraso. O repositório combina informação das tabelas partilhadas de verificação com perfis da JustCauses para comparar dados declarados no perfil com dados devolvidos pelo fornecedor externo. No fluxo KYB, os endpoints de organizações permitem consultar perfis coletivos e submissões documentais, enquanto as decisões exigem \texttt{view\_kyb}. Aprovar ou rejeitar uma submissão atualiza o estado da organização, cria notificação interna, pode enviar e-mail e regista auditoria com \texttt{KYB\_APPROVED} ou \texttt{KYB\_REJECTED}.

\subsection{E-mail transacional e notificações}
\label{subsec:impl_email_notifications}

As notificações por e-mail são acionadas por decisões administrativas relevantes. O operador não envia estes e-mails manualmente; a decisão no backoffice desencadeia uma chamada no \textit{back-end}, que valida a operação, persiste a alteração, regista auditoria e solicita o envio através do Amazon SES. Esta ordem é relevante: a comunicação externa é consequência de uma decisão já validada, não a origem da alteração de estado.

Os e-mails transacionais são construídos por um serviço central de e-mail, com mensagens HTML parametrizadas por dados do domínio, como título da campanha, decisão tomada, nome da organização ou nota administrativa. \section{Implementação do front-end}
\label{sec:impl_frontend}

O \textit{front-end} concretiza a vista de componentes e a vista de processo das Figuras~\ref{fig:c4_level3_fe} e \ref{fig:frontend_process_view}. A interface não é a fonte final de autorização; trabalha sobre o contexto calculado no servidor e torna visíveis as decisões de acesso, página e ação. Por isso, as capturas seguintes não são apresentadas como uma secção isolada de evidências, mas como parte da implementação de cada área funcional.

\subsection{Entrada no ecossistema e acesso por aplicação}
\label{subsec:impl_app_selector}

Depois de iniciar sessão, o operador é encaminhado para o seletor de aplicações. A interface apresenta apenas aplicações incluídas em \texttt{appsAccessible} ou permitidas pelo perfil do operador. Esta decisão concretiza RF03: o acesso ao backoffice administrativo e à aplicação JustCauses é separado por aplicação, antes de qualquer decisão sobre páginas internas.

A Listagem~\ref{lst:frontend_access_rules} mostra como a interface usa o contexto de autorização devolvido pelo servidor para decidir o acesso à aplicação e às páginas. O primeiro bloco corresponde à guarda da JustCauses; o segundo mostra a regra de \textit{page gates} no \textit{layout}. O objetivo desta listagem é responder à ligação entre desenho e implementação: aplicação, página e ação são decisões distintas.

\begin{lstlisting}[float, caption={Decisão de acesso a aplicação e página no front-end}, label={lst:frontend_access_rules}]
const canAccess =
  user?.appsAccessible?.includes("just_causes") ||
  (!user?.roleIsDefault &&
    (user?.roleApplication === "just_causes" ||
      user?.roleApplication === "backoffice"));

function canAccessPage(pageKey: string): boolean {
  if (isSuperAdmin || isBackofficeRole) return true;
  const gate = gates.find((g) => g.pageKey === pageKey);
  if (!gate) return true;
  if (!gate.requiredPermissions.length) return true;
  return gate.requiredPermissions.some((p) =>
    userPerms.includes(p),
  );
}
\end{lstlisting}

A Listagem~\ref{lst:frontend_access_rules} mostra que a interface apenas aplica decisões calculadas a partir do contexto recebido. Se o operador não pertence à aplicação, a guarda apresenta acesso negado antes de renderizar a navegação interna. Se pertence à aplicação mas não tem permissão para uma página, os \textit{page gates} impedem a navegação para essa área. A validação final continua no servidor, através da Listagem~\ref{lst:require_permission}.

test('deve filtrar por application=just_causes', async ({ request }) => {
  const response =
    await request.get('/api/page-gates?application=just_causes');

  expect(response.status()).toBe(200);
  const gates = await response.json();
  expect(Array.isArray(gates)).toBe(true);

  for (const gate of gates) {
    expect(gate.application).toBe('just_causes');
  }
});

test('a atualizacao deve refletir no GET seguinte', async ({ request }) => {
  if (!gateId) return;
  const newPermissions: string[] = [];

  try {
    await request.put(`/api/page-gates/${gateId}`, {
      data: { requiredPermissions: newPermissions },
    });

    const getResponse =
      await request.get('/api/page-gates?application=just_causes');
    const gates = await getResponse.json();
    const updated = gates.find((g: any) => g.gateId === gateId);

    expect(updated).toBeDefined();
    expect(updated.requiredPermissions).toEqual(newPermissions);
  } finally {
    await request.put(`/api/page-gates/${gateId}`, {
      data: { requiredPermissions: originalPermissions },
    });
  }
});
\end{lstlisting}

O primeiro teste da Listagem~\ref{lst:qa_page_gate_tests} confirma que o filtro por aplicação não devolve configurações de outro domínio. O segundo altera temporariamente as permissões exigidas por uma página, valida a persistência através de uma nova leitura e repõe a configuração original no bloco \texttt{finally}. Este detalhe é relevante num ambiente partilhado, porque permite testar persistência real sem deixar dados artificiais na configuração do backoffice.

O relatório de execução Playwright consultado registou 319 testes concluídos com sucesso, sem falhas, erros, testes ignorados ou testes instáveis. Este resultado cobre os testes de \gls{API} e os testes \gls{E2E}. Os testes de integração em Jest/Supertest constam do conjunto de testes disponível e acrescentam 127 cenários definidos, mas não estavam incluídos no relatório Playwright, por serem executados através de um comando próprio.

Não foi disponibilizado um relatório de cobertura de código por linhas ou instruções. Por esse motivo, a avaliação quantitativa apresentada não deve ser lida como uma percentagem formal de cobertura. O que pode ser concluído com rigor é que existe cobertura funcional automatizada sobre módulos sensíveis do backoffice: autenticação transversal, resolução de utilizador autenticado, criação e manutenção de operadores, perfis, permissões, configuração de páginas, logs, audit trail e navegação administrativa.

\section{Sumário}
\label{sec:sumario_cap4}

O trabalho de implementação descrito neste capítulo mostra um backoffice utilizável em contexto operacional e alinhado com o desenho apresentado no Capítulo~\ref{cap:analise_desenho}. A solução integrou a autenticação existente com autorização, auditoria, comunicação externa e administração do acesso: o \textit{front-end} consome o contexto calculado pelo \textit{back-end}, os \textit{page gates} filtram páginas, os endpoints revalidam permissões e os módulos da JustCauses, incluindo campanhas, categorias, \gls{KYC} e \gls{KYB}, demonstram a aplicação prática dessa base comum.
